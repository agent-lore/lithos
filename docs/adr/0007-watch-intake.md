# `WatchIntake` Module: filesystem-driven Corpus mutations as a peer of `CorpusIntake`

ADR-0003 left the file-watcher pipeline on `LithosServer`, with `handle_file_change` and `handle_file_rename` running a similar-but-distinct version of the intake pipeline directly on the server. The deferral was correct given the staged migration shape at the time, but the consequence is that `server.py` is now a second orchestrator alongside `CorpusIntake`: ~60 lines of orchestration plus the watchdog `Observer` lifecycle, the `_FileChangeHandler` adapter, and a `_update_lock` serialisation primitive â€” all justified by an ordering invariant ("emit-before-delete is load-bearing for cache eviction") that further inspection shows the queue-based `EventBus` cannot actually require. `EventBus.emit` delivers events via `asyncio.Queue.put_nowait` to async consumers; no synchronous subscriber path exists, and `EnrichWorker`'s `_resolve_node_id` contract for `NOTE_DELETED` is "use `payload['id']` only; do not check KnowledgeManager because the node has already been deleted." The real invariant on the watcher delete is narrower: `get_id_by_path(relative_path)` must run before `KnowledgeManager.delete(id)` because `delete` clears `_id_to_path`. That is a **capture-before-mutate** invariant local to the caller, not an emit-order invariant. We are extracting the file-watcher pipeline into a `WatchIntake` Module â€” a peer of `CorpusIntake` at `src/lithos/watch_intake.py` â€” that owns three filesystem operations (`upsert_from_disk`, `delete_from_disk`, `rename_on_disk`), the watchdog `Observer` lifecycle, and the serialisation lock. The retraction of the emit-before-delete framing is amended into ADR-0003 in place, with a verification test pinning the corrected ordering.

## Considered Options

- **Wontfix (leave on `LithosServer`).** Rejected â€” `server.py` retains a second orchestrator role with the ordering invariant living as a comment rather than a named seam, and a third caller (bulk import, remote-vault sync) is plausible enough that the cost of having no seam compounds. The deletion test from ADR-0003 ("removing the seam re-spreads the orchestration") bites less hard here (today's watcher pipeline has one caller, `_FileChangeHandler`), but the invariant-as-comment problem is itself sufficient.
- **Unified `Intake` Module with parametrised mode.** Rejected â€” ADR-0003 already rejected unified `apply(mutation)` for the simpler write-vs-delete case on the grounds that "two methods is the honest shape." The watcher case is *more* divergent: a different `KnowledgeManager` entry point (`sync_from_disk` instead of `create`/`update`), rename as a genuine third operation, no agent. A unified shape is strictly worse than the option ADR-0003 already rejected.
- **Extend `CorpusIntake` with watcher methods** (`sync_from_disk`, `delete_by_path`, `rename`). Rejected â€” dilutes `CorpusIntake`'s identity from "the agent-driven intake seam" to "the intake seam for anything." The agent-driven and filesystem-driven cases share machinery but not contract: the agent path requires `ensure_agent_known` and operates on request data; the filesystem path captures id by path and re-reads from disk. Keeping them as peers makes the symmetry visible without conflating their identities.
- **Two peer Modules + shared `_ViewSync` helper.** Rejected â€” extracts ~4 lines of view-sync per Module at the cost of a third class. YAGNI; the right time to extract a shared helper is when a third driver lands, not pre-emptively.
- **Pipeline-only Module; `Observer` and `_update_lock` stay on server.** Rejected â€” leaves `LithosServer` owning the watchdog `Observer`, `_FileChangeHandler`, and the serialisation lock, with the Module only safe to call from a caller that holds the server's lock. Worse than today: the seam exists but the leaky-abstraction shape persists.
- **Single `file_event(path, kind)` method with a discriminator.** Rejected â€” closest to today's `handle_file_change` shape, but the kind-discriminator hides the very capture and ordering invariants the Module exists to expose. Three explicit methods (`upsert_from_disk`, `delete_from_disk`, `rename_on_disk`) put each operation's semantic in the method name; the `_from_disk` / `_on_disk` suffix carries "the filesystem already changed; we are reconciling derived state."
- **Keep the emit-before-delete invariant conservatively.** Rejected â€” preserves a phantom constraint. `EventBus.emit` is queue-based (`asyncio.Queue.put_nowait`) with no synchronous subscriber path; `EnrichWorker` for `NOTE_DELETED` deliberately reads `payload['id']` only and does not consult `KnowledgeManager`. A defensive emit-before-delete ordering would carry forward a constraint that has no consumer â€” exactly the kind of fossilised invariant extraction is supposed to remove. Aligning the watcher delete with `CorpusIntake.delete`'s emit-after-mutate order collapses the only material delete-side asymmetry between the two Modules.
- **Peer Module owning pipeline + watchdog lifecycle + lock, with retracted ordering framing.** Accepted.

## Consequences

- `WatchIntake` lives at `src/lithos/watch_intake.py`. Constructor: `(knowledge, search, graph, event_bus, watch_path)`. No `coordination` dependency â€” watcher-driven mutations do not register agents because there is no agent. The Module holds a private `_update_lock` (relocated from `LithosServer`) that wraps the pathâ†’id capture step on `delete_from_disk` and serialises in-corpus renames on `rename_on_disk`.
- **Three explicit methods:**
  - `async upsert_from_disk(path: Path) -> None` â€” runs `sync_from_disk`, re-indexes the document, notifies the link graph, emits `NOTE_CREATED` or `NOTE_UPDATED`. Order matches `CorpusIntake.write`'s view-sync sequence.
  - `async delete_from_disk(path: Path) -> None` â€” captures id via `get_id_by_path` before any mutation (the actual load-bearing invariant), calls `KnowledgeManager.delete(id)`, removes from Search, notifies graph, emits `NOTE_DELETED` after both view syncs have been kicked off. Matches `CorpusIntake.delete`'s emit-after-mutate order; the previously-claimed emit-before-delete invariant is retracted.
  - `async rename_on_disk(src: Path, dest: Path) -> None` â€” three-way branch preserved from `handle_file_rename`: in-corpus rename (re-binds doc id under the new path via `sync_from_disk` per #202, emits `NOTE_RENAMED`); source leaves corpus (degrades to `delete_from_disk`); destination enters corpus (degrades to `upsert_from_disk`); both outside is a no-op.
- **Lifecycle ownership.** `WatchIntake.start(loop: asyncio.AbstractEventLoop)` creates a private `_FileChangeHandler`, spins up the watchdog `Observer`, schedules it on `watch_path` recursively, and starts it. `WatchIntake.stop()` stops the Observer and joins. `LithosServer.start_file_watcher` / `stop_file_watcher` / `handle_file_change` / `handle_file_rename` / `_FileChangeHandler` / `_observer` / `_watch_loop` / `_update_lock` are deleted. `LithosServer.shutdown()` calls `self.watch_intake.stop()`; the `graph.save_cache()` force-flush that used to live in `stop_file_watcher` migrates to `LithosServer.shutdown` because it is a graph-cache concern, not a watcher concern.
- **Agent attribution.** Watcher-emitted events carry `agent="watcher"` (a system-reserved sentinel). Today's `agent=""` is a magic value distinguishing watcher events from agent events negatively; the sentinel makes the distinguisher affirmative. No subscriber filters on `event.agent` today (it surfaces only in telemetry logs and SSE payloads), so the change is backward-compatible at the consumer surface.
- **Retraction of ADR-0003's emit-before-delete framing.** ADR-0003's final Consequences bullet stated: *"emit-before-delete is load-bearing for the file-watcher case and would need to be expressed at the seam."* Inspection of `EventBus.emit` (events.py:104â€“162) and the two extant subscriber paths (`EnrichWorker.subscribe` in lcma/enrich.py:342, the SSE event stream in server.py:507) shows that subscriber delivery is uniformly `asyncio.Queue.put_nowait`-based â€” there is no synchronous subscriber path, and no subscriber can observe pre-delete `_meta_cache` state regardless of emit ordering. The actual invariant the watcher delete enforces is **capture-before-mutate**: the pathâ†’id resolution must precede `KnowledgeManager.delete` because `delete` clears `_id_to_path`. ADR-0003's Consequences bullet on the file-watcher path is amended in place to retract the emit-order claim and point to this ADR for the corrected framing.
- **Verification test.** The implementation slice ships a regression test that wires a subscriber to `NOTE_DELETED`, triggers a watcher delete, and asserts the subscriber consumes the event with a valid `payload["id"]` while `KnowledgeManager` already reports the id as removed (`get_id_by_path` returns `None`, `_meta_cache` evicted). Guards the corrected ordering against future regression and documents the active subscriber contract.
- **Implementation: one-shot.** Unlike ADR-0003's staged delete-first/write-second migration, the watcher path has a single caller (`_FileChangeHandler`) and the migration is mechanical. A staged split (pipeline-first / lifecycle-second) would create an intermediate state where `WatchIntake` exists but `LithosServer` still holds the watchdog `Observer` and `_update_lock` â€” exactly the leaky-abstraction shape rejected in the Considered Options above. One PR delivers the Module, the deletions on `LithosServer`, the `agent="watcher"` sentinel, the `graph.save_cache` relocation, the ADR-0003 amendment, and the verification test.
- **Out of scope for this ADR.** Dropping `_update_lock` entirely â€” replacing serialisation with idempotency under benign-race semantics â€” is a plausible follow-up but requires a separate analysis of duplicate-event tolerance across subscribers. Surfacing the `agent="watcher"` sentinel in `lithos_events` filtering or SSE client documentation is a follow-up documentation update, not a contract change.
