# Provenance projection becomes its own Module, completing ADR-0001 step 3

`edges.db` and the `_project_provenance_to_edges` function lived under `lcma/edges.py` as one peer module among many in the LCMA package, with no Module-level interface around them. The only path that mutated edges was `reconcile.py:_reconcile_provenance_projection`, which constructed an ephemeral `EdgeStore`, called the projection function as a free helper, and discarded the store — a one-shot operation, structurally different from the plan/apply shape the indices already use after ADR-0001. Scouts that needed to read edges constructed their own `EdgeStore` instances. There was no place to put ADR-0001's step-3 private plan/apply pair, because there was no Module to put it on. We're promoting the provenance projection into its own Module: `src/lithos/provenance.py` exposes `ProvenanceProjection`, hides `EdgeStore` and the projection function as package-internal under `lcma/edges.py`, and provides a private plan/apply pair callable only from `KnowledgeManager`. The Module owns the existence and all columns of its rows; the predicate scoping plan/apply to corpus-derived edges (today `provenance_type="frontmatter"`, expanding as the projection grows) is a private invariant of the Module, not part of the interface. Construction follows ADR-0002's eager-init pattern (`async def create(config)`).

## Considered Options

- **Status quo (free helper called from `reconcile.py`).** Rejected — passes the deletion test as shallow: removing the helper would just require its callers to know how to construct an `EdgeStore` and run the projection. With ADR-0001 step 3 looming and the projection appearing as a derived view in CONTEXT.md, the absence of a Module is what's blocking reconciliation reaching parity with the indices.
- **Add `plan_reconcile_to / apply_reconcile` to `EdgeStore` directly.** Rejected — `EdgeStore` is a SQLite wrapper that today knows nothing about the **Corpus** shape; it would need to import `KnowledgeManager` types, conflating "store" with "projection-of-corpus". Better to keep the store as an internal implementation detail and let the projection Module own the corpus-shaped logic.
- **Fold the projection into `KnowledgeManager`** (mirroring how reconciliation orchestration was folded in ADR-0001). Rejected — `KnowledgeManager` would absorb edge-derivation logic, edge-store schema awareness, and the predicate that scopes plan/apply. That dilutes its identity as the manager of the **Corpus**. Each derived view stays its own Module; the manager coordinates.
- **Module facade in `src/lithos/provenance.py`, `EdgeStore` and projection function private to `lcma/edges.py`, plan/apply package-private (callable only from `KnowledgeManager`).** Accepted.

## Consequences

- `src/lithos/provenance.py` introduces `ProvenanceProjection` with `async create(config)`, public `list_edges(filter)` (and the read methods scouts need today), package-private `_plan_reconcile_to(docs) -> ProvenancePlan` and `_apply_reconcile(plan) -> ProvenanceResult`, and `async close()` for lifecycle.
- `EdgeStore` and `_project_provenance_to_edges` stay in `lcma/edges.py` but stop being imported from outside `provenance.py`. Tests that today reach into `EdgeStore` directly migrate to assertions through `ProvenanceProjection`.
- `KnowledgeManager.ReconcilePlan` (`knowledge.py:625`) gains a `provenance: ProvenancePlan | None` field. `plan_reconcile()` accepts a `projection` argument; `apply_reconcile()` dispatches both. `reconcile.py:_reconcile_provenance_projection` (lines 370–428) goes away — its callers route through `KnowledgeManager.apply_reconcile(plan)`.
- The plan/apply predicate is owned by the Module, not the caller. Today it scopes to `provenance_type="frontmatter"`; as the projection grows to mirror more frontmatter-declared lineage (e.g. `derived_from_ids → type="derived_from"`), the predicate evolves internally. Callers never see the predicate.
- **Invariant**: the projection owns the existence and all columns of its rows. Nothing outside the projection writes to those rows. Annotations, weight tweaks, or evidence text from elsewhere (agents, the enrichment worker) live on **separate asserted edges** with their own `provenance_type`, never as column edits to projection rows. This matches the design doc rule (`docs/plans/lcma-design.md` §1.1: "If frontmatter and `edges.db` diverge, frontmatter is authoritative and projection must be repaired/rebuilt") and keeps the corpus-as-source-of-truth story intact.
- Mutation surface stays **reconcile-only**, matching today's behaviour. `intake.py` does not write to the projection. If incremental per-write edge updates become desired later (parity with search/graph), that's a separate ADR — it would change runtime semantics, not just shape.
- `IndexableDocument` is to `SearchEngine` what `ProvenancePlan` is to this Module: a small value type that crosses the seam. Callers never see SQL rows, never see `EdgeStore`, never see the projection function.
- A new test (`tests/test_provenance_projection.py`) covers plan/apply round-trip, predicate-scoping (an asserted edge with `provenance_type != "frontmatter"` survives reconcile while a stale `wiki_link` edge is deleted), and `KnowledgeManager.apply_reconcile` integration. Existing tests that reach into `EdgeStore` directly are rewritten through the public surface as part of the same change.
- This ADR completes step 3 of ADR-0001's staged migration. With indices (step 1) landed and graph (step 2) pending, `reconcile.py` shrinks further; once step 2 also lands, `reconcile.py` collapses to "loop over views, call plan, call apply" across all three derived views.
