# Cognitive memory becomes its own Module, decoupling `server.py` from LCMA internals

The agent-facing surface of LCMA — `lithos_retrieve`, `lithos_edge_upsert`, `lithos_edge_list`, `lithos_node_stats`, the reinforcement tools, the enrichment worker — was constructed and stitched together at the MCP-server level. `server.py:initialize()` opened a `StatsStore`, opened an `EdgeStore`, started an `EnrichWorker`, imported scout functions, the retrieve orchestrator, and free reinforcement helpers. ~15 call sites across `server.py` reached into these LCMA internals directly. The lifecycle invariants (e.g. "stores must be opened before `EnrichWorker.start()`") lived in inline comments. There was no Module to put a health signal on, no place to centralise the error contract for retrieve (today scout failures are caught-and-logged inline with no shared shape), and no seam for a future caller (bulk import, replication, a non-MCP entry point) to consume LCMA without re-discovering all the wiring. Apply the deletion test: removing the implicit "LCMA package boundary" wouldn't concentrate complexity, because the boundary is shallow — `server.py` is already the orchestrator. The deletion test against a *real* cognitive memory Module fails: ~15 call sites would re-spread the same store-stitching, lifecycle, and error handling. We're extracting `CognitiveMemory` into `src/lithos/cognitive_memory.py`, which owns the StatsStore lifecycle, the scouts, the retrieve orchestrator, the reinforcement methods, the `EnrichWorker`, and the receipts/working-memory tables. It depends on `KnowledgeManager`, `SearchEngine`, `KnowledgeGraph`, `ProvenanceProjection` (per ADR-0004), and `EventBus`. Construction follows ADR-0002's eager-init pattern (`async def create(...)`).

## Considered Options

- **Status quo (server-level stitching).** Rejected — the ~15-call-site coupling is a clear deletion-test failure once the cognitive memory boundary is named honestly. Without a Module, ADR-0001's step-3 plan/apply for the projection has nowhere clean to land, scouts that need to read edges keep constructing their own `EdgeStore`, the `EnrichWorker` lifecycle stays implicit, and the retrieve error contract stays a per-handler concern.
- **One LCMA module covering both projection and cognitive memory.** Rejected — the two have different reconcile semantics. The projection is corpus-derived and has a **Drift** condition; cognitive memory's accumulated state (receipts, salience, working memory, asserted edges) does not. Bundling them dilutes what `plan_reconcile` means and forces one interface to express two unrelated invariants. ADR-0004 takes the projection; this ADR takes the agent-facing surface.
- **Keep methods as free functions in `lcma/*.py`, just add a thin facade.** Rejected — the StatsStore lifecycle, EnrichWorker management, and the error contract genuinely want one owner. A facade without state-ownership reproduces today's coupling under a new name.
- **`CognitiveMemory` Module owning StatsStore, scouts, retrieve, reinforcement, EnrichWorker; `lcma/` stays as the implementation directory.** Accepted.

## Consequences

- `src/lithos/cognitive_memory.py` introduces `CognitiveMemory` with `async create(config, knowledge, search, graph, projection, event_bus)`, public `retrieve(...)`, `node_stats(...)`, `edge_upsert/list/delete(...)`, `reinforce_cited/ignored/between(...)`, `cache_lookup(...)` and `conflict_resolve(...)` where applicable, and a `health() -> Healthy | Unhealthy(reason)` signal mirroring `SearchEngine` per ADR-0002. Lifecycle: `async start()` (starts `EnrichWorker`) and `async stop()` (stops the worker, closes `StatsStore`).
- `lcma/` keeps its files (stats, edges, scouts, retrieve, enrich, reinforcement) as the implementation directory. Imports from outside `lithos.cognitive_memory` and `lithos.provenance` into `lithos.lcma.*` are forbidden after migration. A guard test (or a ruff custom rule) enforces this so re-leakage doesn't happen as the codebase grows.
- `LithosServer.initialize()` constructs in this order: `KnowledgeManager` → `SearchEngine.create(...)` → `KnowledgeGraph` → `ProvenanceProjection.create(...)` → `CorpusIntake(...)` → `CognitiveMemory.create(...)` → `await self.memory.start()`. `shutdown()` reverses: `await self.memory.stop()` first, then close the rest. The implicit lifecycle invariants from today's inline comments become explicit constructor ordering.
- MCP tool handlers across `lithos_retrieve`, `lithos_edge_*`, `lithos_node_stats`, `lithos_reinforce_*`, `lithos_cache_lookup` collapse to one-line wrappers calling `self.memory.<method>` and shaping the envelope. ~15 direct `lithos.lcma.*` imports leave `server.py`. The size reduction is real but secondary; the win is the named seam.
- `errors.py` gains `CognitiveMemoryError(LithosError)` plus specific subtypes (`ScoutFailure` for backend failures the scouts today swallow, and any timeout/retry types the retrieve path needs). `CognitiveMemory.retrieve` documents which exceptions propagate vs which are caught-and-logged in one place. The implicit error contract becomes explicit.
- The reinforcement helpers in `lcma/reinforcement.py` become methods on `CognitiveMemory` rather than free functions taking decomposed stores. `reinforce_cited(receipt_id, cited_ids)` takes the receipt id from a prior `retrieve(...)` call and looks up the relevant nodes through the Module's own state. Free-function shape was a side-effect of the stores being passed around individually; now they aren't.
- **Anticipated evolution (MVP2/MVP3)**: when the enrichment worker materialises concept notes (per `lcma-design.md` §1.1) or writes back LLM-generated summaries, those mutations flow through `CorpusIntake.write` with a synthetic `agent_id="lithos-enrich"`. `CognitiveMemory` will gain a `CorpusIntake` dependency at that point — directional, no cycle, one-line constructor change. ADR-0005 places the seam now so the dep drops in cleanly later. MVP1 does not need it.
- **Anticipated evolution**: the MVP2 extension to `lithos_task_complete` (which passes `cited_nodes` / `misleading_nodes` and triggers reinforcement alongside coordination updates) makes that handler an explicit multi-Module orchestrator — it calls `coordination.complete_task(...)` and then `cognitive_memory.reinforce_*(...)`. Two short calls, order documented; no new abstraction unless a third such tool emerges.
- **Anticipated evolution**: scouts that today read private `KnowledgeManager` state (e.g. the MVP2 source-url scout reaching into `_source_url_to_id`) get migrated to public `KnowledgeManager.find_by_source_url(...)`-shaped methods as part of their own move into the Module. Same pattern as the broader "interface is the test surface" cleanup.
- New tests (`tests/test_cognitive_memory.py`) cover Module construction, retrieve through the seam, `reinforce_cited` updating stats, and `EnrichWorker` lifecycle (`start` then `stop` does not leak threads). Pure unit tests of `StatsStore`, `EdgeStore`, and individual scouts stay where they are — they're testing implementation details, which is fine for internal-only code.
- `lithos_edge_upsert` ships on `CognitiveMemory` wrapping today's behaviour. ADR-0006 may relocate it to `CorpusIntake.assert_edge` once the source-of-truth question for asserted edges is resolved; the wrapping makes that move a single-method change.
