# `edges.db` is authoritative for agent-asserted edges; broaden Corpus to acknowledge it

CONTEXT.md defines the **Corpus** as "the set of Markdown notes on disk … treated as the single source of truth." ADR-0004 establishes the **Provenance projection** as a derivable view of the Corpus — corpus-derived edges (today `provenance_type="frontmatter"`) live in `edges.db` and are repaired by `Reconcile`. But `edges.db` also holds **agent-asserted edges**: rows written by `lithos_edge_upsert` carrying types like `related_to`, `supports`, `contradicts`, `is_example_of`, `depends_on`, `analogy_to` (and in MVP2, `conflict_state` for the contradiction workflow). These rows have no representation in any Markdown note. They live only in `edges.db`. Under today's CONTEXT.md they are anomalies — agent-authored persistent state that sits outside the source of truth. Per the LCMA design doc (`docs/plans/lcma-design.md` §1.1, §3.1, line 113-117), this is by design: `edges.db` is authoritative for asserted-edge content; only the lineage subset (`type="derived_from"`, frontmatter-mirrored) is treated as a projection of the Corpus. CONTEXT.md hasn't caught up with that commitment, and routing `lithos_edge_upsert` through `CorpusIntake` (the agent-driven mutation seam from ADR-0003) would today involve a write surface intake doesn't currently understand. We're broadening **Corpus** in CONTEXT.md to acknowledge two storage tiers — Markdown notes (note content + frontmatter-declared lineage) and `edges.db` (asserted edges) — and extending `CorpusIntake` with an `assert_edge` method whose invariants (atomicity, view sync, event emit on success, no event on view-sync failure) mirror `write` and `delete`.

## Considered Options

- **Persist agent-asserted edges to disk as new file types** (e.g. a `data/edges/<id>.json` directory, special "edge declaration" notes, or frontmatter-list extensions on related notes). Rejected — invents a new on-disk format the design doc explicitly rejects (`edges.db` is described as the storage for non-derived edges; `lcma-design.md` §4.1 lists `edges.db` under `data/.lithos/` as the canonical location). Adds migration cost and a parallel concept ("edges as files" vs "edges as rows") with no reader benefit. Discarding the proposal preserves the design doc's choice of SQLite for fast weight/stat updates without rewriting Markdown.
- **Leave `lithos_edge_upsert` outside CorpusIntake, on Cognitive memory.** Rejected — works mechanically (and is the interim state ADR-0005 ships) but leaves the agent-driven mutation pipeline split into two unrelated paths: notes through intake with `expected_version`, atomicity, view sync, event emit; edges through Cognitive memory with ad-hoc transaction boundaries, no event-after-views invariant, no synthetic-actor accounting. The split bakes in two contracts for "agent writes a thing" forever.
- **Forbid agent-asserted edges entirely** (force every relationship into derivable form via wiki-links or `derived_from_ids` frontmatter). Rejected — collapses real expressiveness. The design doc carries `supports`, `contradicts`, `analogy_to`, `weight`, `evidence`, `conflict_state` as core MVP2/MVP3 vocabulary. Demanding agents synthesise all of that as text in note bodies just to round-trip through projection is a worse interface than letting agents call `assert_edge` directly.
- **Broaden the Corpus definition; add `CorpusIntake.assert_edge`.** Accepted (proposed pending review).

## Consequences

- CONTEXT.md's **Corpus** definition expands to "the set of Markdown notes on disk under the configured data directory, **and** the agent-asserted edges in `data/.lithos/edges.db`. Both are agent-authored persistent state and the joint source of truth. Derived views (Search engine, link graph, Provenance projection) are derivable from this joint source." The projection-vs-asserted split inside `edges.db` is preserved by the projection's predicate (per ADR-0004): derived-edge rows are owned by the projection, asserted-edge rows are owned by intake.
- `CorpusIntake` (per ADR-0003) gains a third method: `async assert_edge(agent, request) -> EdgeOutcome`. The pipeline is **four steps**, not five — there is no view-sync step, because no edge-derived view exists today. If a future incremental edge-derived view appears, it lands via its own ADR.
  - Ensure agent registration (`ensure_agent_known`, unconditional, matching `write` / `delete`).
  - Mutate `edges.db` atomically (single transaction).
  - Emit `EDGE_UPSERTED` on success (already a planned event type per the LCMA MVP1 prerequisites). Emit failures stay non-propagating, same as `NOTE_*`.
  - No `expected_version` semantics: `lithos_edge_upsert` is upsert-by-natural-key (`from_id, to_id, type, namespace`); concurrent edits are not a known caller need. If they become one, address with a separate ADR.
- **Migration is staged across three slices.** **Slice 1 (this ADR):** the `lithos_edge_upsert` MCP handler routes through `CorpusIntake.assert_edge`; `CognitiveMemory.edge_upsert` becomes a one-call wrapper. **Slice 2 (future ADR):** `EnrichWorker`'s `consolidation`-typed edges flow through `assert_edge` with synthetic actor `lithos-enrich`; includes review of self-event-loop risk on `EDGE_UPSERTED` subscribers. **Slice 3 (future ADR):** reinforcement writes flow through intake; resolves the `EdgeStore.adjust_weight` shape question (new intake method? stay direct as an asserted-edge implementation detail?). Without these stages, "asserted edges flow through intake" is an aspiration rather than an enforced contract.
- **`EdgeStore` becomes a public peer Module** (importable from `provenance.py` and `intake.py`). This **partially supersedes** ADR-0004's clause that `EdgeStore` not be imported outside `provenance.py` — that clause was an implementation-detail tightening of ADR-0004's load-bearing invariant ("the projection owns the existence and all columns of its rows; nothing outside the projection writes to those rows"), which survives unchanged. The row-ownership invariant is now jointly enforced by `provenance_type` predicate-scoping in both `ProvenanceProjection` (writes only `frontmatter` rows during reconcile) and `CorpusIntake` (writes only non-projection rows during `assert_edge`). `server.py:initialize()` constructs `EdgeStore` directly and passes the same instance to `ProvenanceProjection.create(...)` and `CorpusIntake(...)`, replacing today's `self.edge_store = self.projection._edge_store` reach-through.
- **Read APIs unchanged.** `ProvenanceProjection.list_edges` continues to serve both projection-class and asserted-class rows; reads have no intake invariants to honor, so the broadened **Corpus** does not affect the read path.
- `CognitiveMemory.edge_upsert` (from ADR-0005) becomes a one-call wrapper that builds the request and calls `intake.assert_edge(agent, request)`. The MCP tool handler stays one-line on top of cognitive memory. The migration is local: one method on intake, one rewrite on cognitive memory, zero change at the MCP surface.
- `CognitiveMemory` gains a `CorpusIntake` dependency at this point (a precursor to the same dependency MVP2/MVP3 needs for enrich-worker concept materialisation, per ADR-0005's anticipated evolution). The dep is directional and acyclic.
- The "synthetic actor" concept (e.g. `agent_id="lithos-enrich"`) generalises: any non-human writer — the enrichment worker, a future replication agent, a bulk import — registers as an actor with `CoordinationService` and writes through intake. The actor name appears in `provenance_actor` on edges and in event payloads, giving asserted-edge writes the same audit trail as note writes.
- **Out of scope for this ADR**: incremental edge-store reconciliation (today edges only update via `Reconcile`, even after intake writes them — see ADR-0004). Asserted-edge writes are immediately durable in `edges.db` because `assert_edge` is an atomic mutation of the source-of-truth tier; no separate "view sync" needs to happen. Derived edges are still reconcile-only.
- **Out of scope for this ADR**: `lithos_conflict_resolve`. It writes to `edges.db` (via `update_conflict_resolution`) and notes (via `knowledge.update(supersedes=...)`) in a single handler, and is not relocated by this ADR. It is a multi-Module orchestrator (per ADR-0005's anticipated pattern) with a cross-storage atomicity question that requires its own design — folding it into intake means either introducing a cross-storage transaction primitive or splitting it into two intake calls with a documented ordering. Deferred to a follow-up ADR. Without this note, `lithos_conflict_resolve` reads as an inconsistency with the broadened **Corpus** rather than as deferred scope.
- This ADR was originally marked `proposed` because it broadens a load-bearing definition (**Corpus**); it is now `accepted` after the HITL grilling tracked in issue #252. ADR-0005 shipped independently with `lithos_edge_upsert` on cognitive memory; the relocation to intake lands in Slice 1 of the staged migration above.
